## 2. ช่องทางการรับข้อมูล OPPP Individual

```
หน่วยบริการ (HIS) ──→ สป.สธ. (HDC) ──→ สปสช. (OPPP individual)
                                            ↑
หน่วยบริการประจำ (รพ.) ──→ สสจ. ──(1)──→ สปสช.
                           ↓
                    ──────(2)──────────→ สปสช.
```

**2 ช่องทาง:**
1. ผ่าน สสจ. (สำนักงานสาธารณสุขจังหวัด)
2. ส่งตรงจาก รพ. ไปยัง สปสช.

### กรอบเวลาการรับข้อมูล

สำหรับ OPPP individual data:
- **date_serv** (วันที่รับบริการ): ต.ค. ปีก่อน – ก.ย. ปีงบ
- **date_send** (วันที่ส่ง): พ.ย. ปีก่อน – ต.ค. ปีงบ
- รับข้อมูลตามโครงสร้างมาตรฐาน 43 แฟ้ม V 2.3

## 3. การเชื่อมโยงแฟ้มเพื่อประมวลผล

### แฟ้มบริการ OP (ผู้ป่วยนอก)

```
PERSON* ──→ SERVICE ──→ DIAGNOSIS_OPD ──→ DRUG_OPD
(HOSPCODE,    (HOSPCODE,    (HOSPCODE,       (HOSPCODE,
 PID,          PID,          PID,             PID,
 CID)          SEQ,          SEQ,             SEQ,
               DATE_SERV)    DATE_SERV)       DATE_SERV)
                                           ──→ PROCEDURE_OPD (Add on)
                                              (HOSPCODE, PID, SEQ, DATE_SERV)
                                           ──→ CHARGE_OPD
                                              (HOSPCODE, PID, SEQ, DATE_SERV)
```

**Key fields สำหรับเชื่อมโยง:** HOSPCODE, PID, SEQ, DATE_SERV
**ระบบเปิดรับโครงสร้าง version 2.3**

### แฟ้มบริการ PP (ส่งเสริมป้องกัน)

```
PERSON* ──→ ANC ──→ PRENATAL
         ──→ NUTRITION
         ──→ LABFU
         ──→ NCDSCREEN
         ──→ SPECIALPP
         ──→ CHRONICFU
         ──→ FP
         ──→ EPI
         ──→ LABOR
         ──→ NEWBORN
```

แต่ละแฟ้ม PP ใช้ key fields: HOSPCODE, PID, SEQ, DATE_SERV

### แฟ้มสะสม (Cumulative)

```
PERSON* ──→ CHRONIC (HOSPCODE, PID)
         ──→ DRUGALLERGY (HOSPCODE, PID)
         ──→ DISABILITY (HOSPCODE, PID)
```

**กฎ:**
- ส่งพร้อมแฟ้ม PERSON ทุกครั้ง
- ส่งทั้งหมดได้ ในครั้งเดียว และส่งส่วนที่เพิ่มเติมในเดือนถัดไป
- ยกเว้นแฟ้ม PERSON ไม่ได้รับทั้งหมด ต้องส่งมาพร้อมแฟ้มบริการ ทุกครั้ง

## 4. กระบวนการตรวจสอบข้อมูล

### 3 ขั้นตอน:

#### ขั้นตอนที่ 1: ตรวจสอบไฟล์ข้อมูล (Validate)
ตรวจสอบ:
- ชื่อไฟล์ถูกต้อง
- Format ถูกต้อง
- Unzip ได้
- ชื่อไฟล์ไม่ซ้ำ
- ไม่มี special character
- ตรวจความถูกต้อง data type ตามโครงสร้างข้อมูล

**ถ้าไม่ผ่าน:** รูปแบบไฟล์ไม่ถูกต้อง → แก้ไขให้ถูกต้อง ส่งใหม่

#### ขั้นตอนที่ 2: ตรวจสอบข้อมูล (Verify)
- ตรวจความเชื่อมโยงของแฟ้ม
- ตรวจสอบรหัสอ้างอิง ตาม สธ.
- **ตรวจสอบรหัส ICD-10, ICD-9**
- ตรวจสอบซ้ำซ้อน

**ถ้าไม่ผ่าน:** ได้ **ERROR CODE** → แก้ไขแล้วส่งใหม่

#### ขั้นตอนที่ 3: ประมวลผล
- ข้อมูลผ่าน ถูกต้อง
- ข้อมูลไม่ซ้ำซ้อน
→ **นำไปคิดผลงานบริการ**

## 5. แฟ้ม DATA_CORRECT (การแก้ไขข้อมูล)

### เงื่อนไข:
- ต้องเป็น **ข้อมูลที่ผ่านการตรวจสอบ** (ไม่ติด error)
- ทุกครั้งที่มีการแก้ไข ต้องส่งข้อมูลที่แก้ไขมาพร้อมกับแฟ้ม DATA_CORRECT
- ข้อมูลที่ไม่ผ่านการตรวจสอบหรือติด error สามารถส่งใหม่ตามรอบการแก้ไขปกติ
- สปสช. จะถือว่าเป็นข้อมูลที่แก้ไข และ ยึดถือข้อมูล วันที่ทำการส่ง (date_send) การปรับ ณ ครั้งล่าสุด
- กรณีที่ส่งแฟ้ม DATA_CORRECT มาแต่ไม่ได้ส่งข้อมูลที่แก้ไขมาในแฟ้มที่ต้องการแก้ไข → ระบบจะทำการลบข้อมูลตามแฟ้ม DATA_CORRECT ที่ส่งมา

### กระบวนการ:

**แบบ 1:** ส่ง DATA_CORRECT อย่างเดียว (ไม่ส่งข้อมูลแก้ไขมา)
```
ข้อมูล → ข้อมูลผ่าน → DATA_CORRECT → ยกเลิกข้อมูล (Warning!)
```

**แบบ 2:** ส่ง DATA_CORRECT + ข้อมูลที่แก้ไข
```
ข้อมูลแก้ไข + DATA_CORRECT → ข้อมูลผ่าน → ยกเลิกข้อมูลเดิม (Warning!)
                             → ข้อมูลแก้ไข → ข้อมูลใหม่
```

### รอบการประมวลผล DATA_CORRECT
- สามารถแก้ไขได้และส่งภายในรอบก่อนสิ้นเดือนถัดไปของทุกเดือน
- แต่ละรอบข้อมูลบริการ (date_serv) มีช่วงเวลาที่สามารถแก้ไขได้

## 6. กรณีที่สามารถส่งไฟล์เข้ามาใหม่ได้

- **ข้อมูลบริการที่ยังไม่ได้ส่ง** ในปีงบประมาณ หรือ ย้อนหลัง 1 ปีกรณีที่ไม่ได้ส่ง
- ข้อมูลที่ **แก้ไข error** แล้วส่งใหม่
- ข้อมูลที่ติด error **(XX9299) ซ้ำซ้อน** ไม่ต้องส่งใหม่ **ยกเว้นกรณี ANC, PAP, TTM** (ระบบ OPPP ไม่มีระบบอุทธรณ์)
- ข้อมูลที่ส่งแล้วแต่ระบบแจ้งว่า **"text file ไม่ถูกต้อง"**

## ข้อสำคัญสำหรับ AI Claim Validator

1. **Key fields ที่ต้องตรวจ:** HOSPCODE, PID, CID, SEQ, DATE_SERV
2. **CID Not Match 3.38%** — ต้องตรวจสอบเลข 13 หลักให้ตรง
3. **Error จากชื่อไฟล์ผิด** — ตรวจ format ชื่อไฟล์ (F43_XXXXX_YYYYMMDDHHMMSS.zip)
4. **ICD-10/ICD-9 validation** เกิดขึ้นในขั้นตอน Verify (ขั้นตอนที่ 2)
5. **DATA_CORRECT** ต้องส่งคู่กับข้อมูลที่แก้ไข มิฉะนั้นข้อมูลเดิมจะถูกลบ
6. **ข้อมูลซ้ำซ้อน (XX9299)** ไม่ต้องส่งใหม่ ยกเว้น ANC, PAP, TTM
7. **ส่งพร้อมแฟ้ม PERSON ทุกครั้ง**
