# Zero C — ระบบจ่ายศูนย์บาทรายไอเทม (แทน C ทั้งเคส)

## Definition

**Zero C** คือนโยบายและระบบใหม่ของ สปสช. ปีงบประมาณ 2569 ที่เปลี่ยนวิธีจัดการข้อมูลการเบิกจ่ายที่ไม่ผ่านเงื่อนไข จากเดิม "ติด C ทั้งเคส — ไม่จ่ายทั้งเคส" มาเป็น "จ่าย **0 บาทเฉพาะรายไอเทมที่มีปัญหา** — รายการอื่นจ่ายตามปกติ"

เป้าหมาย: แก้ปัญหาหน่วยบริการขาดสภาพคล่องจาก C-code backlog และลดภาระการแก้ไขข้อมูลซ้ำซ้อน

## ปัญหาก่อน Zero C (ระบบเดิม)

- หน่วยบริการส่งข้อมูลมาเบิกจ่าย → ติด C-code ข้อหนึ่ง → **ไม่จ่ายทั้งเคส**
- ต้องรอ REP (รายงานตอบกลับรายวัน) ช่วงเย็นหรือวันถัดไป
- ต้องนำข้อมูลไปแก้ไขทั้งหมดแล้วส่งใหม่ — เสียเวลามาก
- บางรายติดเรื่องเล็กน้อย เช่น ราคา 10-20 บาท แต่ทั้งเคสมูลค่าเป็นหมื่น ก็ไม่ได้รับเงินเลย
- มี C-code backlog สะสมหลายปี หน่วยบริการยังไม่ได้รับเงิน

## วิธีใหม่ภายใต้ Zero C

### 1. Near Real-Time Validation

- ส่งข้อมูลเข้ามา → ระบบตรวจสอบ **ทันที** (ไม่ต้องรอตอนเย็น)
- แจ้งผลว่าผ่าน/ไม่ผ่าน พร้อมสาเหตุ แบบ near real-time
- หน่วยบริการแก้ไขได้รวดเร็ว ก่อนจบวันบริการ

### 2. Line Item จ่าย 0 บาท (แทนติด C ทั้งเคส)

- รายการไหนไม่ผ่านเงื่อนไข → จ่าย **0 บาทเฉพาะรายการนั้น**
- รายการอื่นในเคสเดียวกัน → **จ่ายตามปกติ**
- หน่วยบริการได้รับเงินเร็วขึ้น สภาพคล่องดีขึ้น

### 3. รายงาน Data Sheet 0 บาท

- ระบบออก **รายงาน (REP/Data Sheet)** แจ้งว่ารายการไหนจ่าย 0 บาท ด้วยเหตุผลอะไร
- หน่วยบริการสามารถ export ข้อมูลออกมาตรวจสอบได้
- ดูได้จาก **Audit Resource Column** ในหน้าข้อมูลเบิกจ่าย

### 4. การแก้ไขและอุทธรณ์

- ถ้าต้องการแก้ไขรายการที่จ่าย 0 บาท → อุทธรณ์/แก้ไขข้อมูลแล้วส่งใหม่
- ถ้าผ่านเงื่อนไข → จ่ายใน **Statement รอบถัดไป** อัตโนมัติ (ระบบอิเล็กทรอนิกส์ทั้งหมด)

## การปรับตัวของหน่วยบริการ

| เดิม | ใหม่ (Zero C) |
|------|--------------|
| ส่งข้อมูล → รอ REP ตอนเย็น/วันถัดไป | ส่งข้อมูล → เห็นผลทันที |
| ติด C → ไม่ได้เงินทั้งเคส | ติด C รายไอเทม → จ่าย 0 บาทเฉพาะรายการนั้น |
| ต้องแก้ทั้งเคส ส่งใหม่ | แก้เฉพาะรายไอเทมที่มีปัญหา |
| มี C backlog สะสม | ลด backlog, สภาพคล่องดีขึ้น |

**Flow ที่แนะนำ:**
- คีย์อิน: ดู response ของระบบหลังบันทึกทันที → แก้ทันท่วงที
- Bulk upload: ดาวน์โหลดผลตอบรับ → ตรวจสอบ → แก้ไขรายการที่ไม่ผ่าน

## สถานะ C30 Backlog (ณ วันที่ประชุม ม.ค. 2569)

- สปสช. กำลัง **cleansing ข้อมูล** C30 backlog จากหลายปี
- คาดเปิดให้แก้ไข: **ภายในมกราคม 2569** (เร็วสุด ~15 ม.ค.) หรือไม่เกินกุมภาพันธ์ 2569
- จะรวม e-Claim และ non-e-Claim ไว้ในรอบเดียวกัน เพื่อไม่ให้หน่วยทำงานซ้ำซ้อน

## ช่องทางตรวจสอบ

- **ระบบ e-Claim / ec**: หน้าส่งข้อมูล → ดูผลตรวจสอบเบื้องต้นทันที
- **Audit Resource Column**: แสดงรหัส C และ Error ที่ทำให้ไม่ผ่าน
- **EKM Kant / Export Data**: ดาวน์โหลดรายละเอียด line item ออกมาวิเคราะห์
- **Data Sheet 0**: รายงานรายการที่จ่าย 0 บาทพร้อมเหตุผล
